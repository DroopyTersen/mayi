## Overview
Add a reusable `HouseRulesDrawer` that renders the markdown from `docs/house-rules.md` and expose it via a help icon in the game header and a button on the home page. The drawer uses the existing `ResponsiveDrawer` to switch between Dialog (desktop) and Drawer (mobile), ensuring consistent UX and scrollable long content. Both entry points control the same `HouseRulesDrawer` via `open`/`onOpenChange`.

## Key design decisions with rationale
- **Use `ResponsiveDrawer` for layout switching**: keeps mobile/desktop behavior consistent and aligns with existing app patterns.
- **Controlled open state (`open`, `onOpenChange`)**: enables both entry points to manage visibility without internal state, making it composable and testable.
- **Render markdown via `?raw` import**: keeps `docs/house-rules.md` as the single source of truth and avoids duplication.
- **Apply markdown-friendly styling**: use Tailwind prose classes or existing markdown styles to ensure headers/tables/lists render correctly.
- **Scoped `HouseRulesDrawer` in `app/ui/house-rules/`**: aligns with UI component organization and avoids creating barrel files.

## Trade-offs analysis
- **Dev time**: Low. Reuses existing `ResponsiveDrawer` and markdown source; only small UI wiring is required.
- **Risk**: Moderate around markdown rendering if no existing renderer is in use. If none exists, adding a renderer may add a new dependency. If one exists, reuse it to reduce risk.
- **Maintainability**: High. Single source of truth (`docs/house-rules.md`) and a shared UI component reduce duplication.
- **Testability**: Good. Controlled `open` state makes the drawer easy to test with DOM queries; can add a simple render test if needed.
